Документ описывает минимальный market test для **KkalScan**: готов ли пользователь платить или смотреть рекламу за сканирование еды по фото.

## 0. Два вида «рекламы» (не путать)

| | Привлечение (UA) | Монетизация (в приложении) |
|--|----------------|----------------------------|
| **Что** | Как люди **узнают** и ставят приложение | Как пользователь **платит за сканы** после лимита |
| **Инструменты** | ASO RuStore, посты TG/VK, **VK Реклама** → ссылка на RuStore | **Rewarded video** (Яндекс) + подписка 199 ₽ |
| **На тесте** | Сначала **бесплатно**, платно — только если мало установок | **Оба** варианта на экране лимита — иначе не узнаем, что работает |

**Ответ на «только реклама в RuStore»:**

- **Привлечение:** не только платная реклама. Схема: **2 недели бесплатно** (ASO + посты) → если < 50 установок → **VK Реклама** со ссылкой на карточку в RuStore (официальный канал RuStore, без трекера). Баннеры *внутри* магазина — опционально, не обязательны на тесте.
- **Монетизация:** не только реклама — на экране лимита остаются **реклама И подписка**, чтобы проверить обе гипотезы.

## 1. Цель теста

### Главный вопрос

> После того как человек попробовал скан и увидел ценность — он **платит / смотрит рекламу** или **уходит**?

### Что проверяем

| Гипотеза | Критерий успеха |
|----------|-----------------|
| Есть ценность продукта | D7 retention ≥ 15% |
| Пользователь доходит до монетизации | ≥ 25% активных хотя бы раз упираются в дневной лимит |
| Готовность к рекламе | ≥ 15% от `limit_hit` досматривают rewarded video |
| Готовность платить | ≥ 1% установок → старт подписки (или ≥ 0.5% при 100+ установках) |

### Что НЕ проверяем на этом этапе

- Идеальную точность распознавания русской еды
- iOS, Google Play, веб-версию
- Масштабирование и unit-экономику на большой базе
- Конкуренцию с CalZen по MAU

### Решение после теста

| Результат | Действие |
|-----------|----------|
| Реклама ≥ 15% + оплата ≥ 1% | Масштабировать: Google Play, улучшать точность, поднять цену |
| Только реклама работает | Модель ad-first, подписка вторична |
| Только оплата работает | Ослабить рекламу, цена 299 ₽ |
| Retention ок, монетизация слабая | Менять paywall (цена, лимит, момент показа) — второй тест |
| Retention < 10%, монетизация 0 | Продукт не заходит — стоп или pivot |

## 2. Scope продукта (v0.1)

### Must-have

- [ ] Камера → фото → Vision API → блюдо + ккал + БЖУ
- [ ] Дневник питания на сегодня (список приёмов)
- [ ] Лимит **3 скана/день** бесплатно
- [ ] Экран после лимита: **реклама (+2 скана)** или **Pro 199 ₽/мес**
- [ ] Точка эквайринг (подписка 199 ₽ на сайте)
- [ ] Rewarded video ads (Яндекс Реклама / myTarget)
- [ ] Аналитика воронки (AppMetrica)
- [ ] Работа без регистрации (гость по device id)

### Сознательно не делаем

- iOS, Google Play, лендинг, SEO
- Штрихкод, вода, графики, Health / Google Fit
- База русских блюд (только промпт)
- VK-auth, запрос отзыва до первого скана
- Ручная правка граммов (nice-to-have v0.2, не блокер теста)

## 3. Модель монетизации на тесте

```
Каждый день:
  3 бесплатных скана
       ↓ лимит исчерпан
  ┌──────────────────────────────────────┐
  │  Смотреть рекламу 30 сек → +2 скана  │
  │  Pro 199 ₽/мес → безлимит            │
  │  Завтра снова 3 бесплатных           │
  └──────────────────────────────────────┘
```

### Почему так

| Решение | Обоснование |
|---------|-------------|
| 3 скана/день | Хватает оценить ценность (завтрак + обед + ужин); активные упираются в лимит за 1–2 дня |
| 199 ₽, не 299 ₽ | На тесте важен сигнал «готов платить вообще», не максимальный ARPU |
| +2 скана за рекламу | Достаточно для проверки willingness; не раздаём безлимит бесплатно |
| Потолок API 5 000 ₽/мес | Жёсткий лимит в админке / биллинге провайдера — защита бюджета |

| Модель | ₽/скан | 1 000 сканов | 5 000 сканов |

## 6. Технический MVP — задачи (15 шт.)

Оценка для соло: **10–14 рабочих дней**.

### Backend / API (4 задачи)

| # | Задача | Критерий готовности |
|---|--------|---------------------|
| 1 | Прокси-эндпоинт `POST /api/v1/scan` на `91.207.75.72:8080` (multipart) | Фото уходит на сервер, не с устройства напрямую в LLM |
| 2 | Интеграция Vision API (Gemini Flash или аналог) | Ответ JSON: `[{name, grams, kcal, protein, fat, carbs}]` |
| 3 | Промпт под еду + русский язык в ответе | На 5 тестовых фото возвращает осмысленный результат за < 10 сек |
| 4 | Rate limit: 3 скана/день/device_id + `POST /api/v1/scan/bonus` | 4-й скан без рекламы/оплаты → 429 + флаг `limit_hit` |
| 4a | API дневника: `GET/POST/DELETE /api/v1/diary` + SQLite/Postgres в Docker | Записи персистентны на сервере, не на устройстве |

### Android-клиент (6 задач)

| # | Задача | Критерий готовности |
|---|--------|---------------------|
| 5 | Экран камеры / выбор из галереи | JPEG сжатие до ~500 KB перед отправкой |
| 6 | Экран результата скана + добавление в дневник | Пользователь видит ккал и БЖУ, запись в список дня |
| 7 | Дневник «Сегодня» — загрузка с `GET /api/v1/diary`, удаление через API | Список приёмов и итого ккал с сервера |
| 8 | Экран `limit_hit`: кнопки «Реклама» и «Pro 199 ₽» | Показ только после исчерпания лимита |
| 9 | Точка API: страница `http://91.207.75.72:8080/pay`, webhook, `is_pro` на сервере | Оплата 199 ₽ → безлимит сканов для `device_id` |
| 10 | Rewarded ads: показ → callback → +2 скана на сегодня | После досмотра можно сделать ещё 2 скана |

### Инфраструктура и аналитика (3 задачи)

| # | Задача | Критерий готовности |
|---|--------|---------------------|
| 11 | AppMetrica: события из раздела 7 | Все события видны в отчёте воронки |
| 12 | Деплой backend на `91.207.75.72:8080` (Docker, порт 8080) | APK ходит на `http://91.207.75.72:8080/api/v1/` |
| 13 | Потолок расходов API (алерт или hard stop на 5 000 ₽) | При достижении лимита сканы возвращают понятную ошибку |

### RuStore релиз (2 задачи)

| # | Задача | Критерий готовности |
|---|--------|---------------------|
| 14 | Подпись AAB, загрузка в RuStore, карточка приложения | Статус «На модерации» / «Опубликовано» |
| 15 | Проверка на 3 реальных устройствах из РФ | Установка из RuStore → скан → лимит → реклама или покупка |

## 7. События аналитики

### Обязательные события

| Событие | Когда | Зачем |
|---------|-------|-------|
| `install` | Первая сессия | База воронки |
| `first_scan_success` | Первый успешный скан | Activation |
| `second_scan_success` | Второй скан | Повторное использование |
| `third_scan_success` | Третий скан | До лимита |
| `limit_hit` | 4-й скан заблокирован | **Ключевое** — дошли до монетизации |
| `scan_error` | Скан завершился ошибкой | Диагностика качества распознавания / сети / backend |
| `photo_picker_cancel` | Фото не выбрано | Отмена, отказ camera permission или сбой picker |
| `add_to_diary_failed` | Не удалось сохранить результат | Диагностика diary/API после успешного скана |
| `ad_offer_shown` | Показан экран с рекламой | |
| `ad_bonus_click` | Нажали rewarded video | |
| `ad_watch_complete` | Досмотрели rewarded video | **Готовность к рекламе** |
| `ad_bonus_failed` | Ошибка выдачи бонуса | |
| `paywall_shown` | Показана кнопка Pro | |
| `paywall_back` | Ушли с paywall назад | Качество оффера / цена / доверие |
| `pro_click` | Нажали кнопку Pro | Намерение купить |
| `subscription_start` | Успешная покупка | **Готовность платить** |
| `subscription_cancel` | Отмена (если видна) | Качество удержания |
| `day_1_return` | Вернулся на следующий день | Retention |
| `day_7_return` | Вернулся через 7 дней | Retention |

### Формулы

```
Готовность к рекламе = ad_watch_complete / limit_hit
Готовность платить   = subscription_start / install
Доходят до paywall   = limit_hit / third_scan_success
Ошибки скана         = scan_error / photo_scan
Сохранение дневника  = add_to_diary / scan_success
```

### Минимальная сводка (неделя 6)

| Метрика | Значение | Порог |
|---------|----------|-------|
| Установки | ? | ≥ 100 |
| first_scan_success / install | ?% | ≥ 40% |
| limit_hit / активные (≥2 скана) | ?% | ≥ 25% |
| ad_watch_complete / limit_hit | ?% | ≥ 15% |
| subscription_start / install | ?% | ≥ 1% |
| day_7_return | ?% | ≥ 15% |

## 8. RuStore: регистрация и монетизация (пошагово)

### 8.1. Юридическое лицо

**Статус: ИП уже есть** — шаг пропускаем.

Осталось:
- [ ] Привязать реквизиты ИП в кабинете RuStore
- [ ] Расчётный счёт для выплат (если ещё не привязан к RuStore)

### 8.2. Аккаунт разработчика RuStore

1. Перейти на [console.rustore.ru](https://console.rustore.ru) (или актуальный URL кабинета разработчика).
2. Войти через VK ID / телефон.
3. Заполнить профиль разработчика: имя, контакты, реквизиты ИП/самозанятого.
4. Принять оферту разработчика.
5. Дождаться верификации (обычно 1–3 рабочих дня).

**Стоимость:** 0 ₽.

### 8.3. Создание приложения

1. «Создать приложение» → тип: приложение.
2. Название: `KkalScan — калории по фото бесплатно` (см. [brand-dns.md](./brand-dns.md)).
3. Категория: **Здоровье** или **Продуктивность**.
4. Загрузить иконку 512×512, скриншоты (минимум 2).
5. Краткое и полное описание (ключи из раздела 9).
6. Политика конфиденциальности — `http://91.207.75.72:8080/privacy` (или GitHub Pages, если RuStore не примет HTTP).

### 8.4. Подпись и загрузка APK/AAB

1. Создать keystore для release-сборки (хранить backup — потеря = нельзя обновлять приложение).
2. Собрать signed AAB.
3. В кабинете: версия → загрузить AAB → указать `versionCode`, `versionName`.
4. Пройти проверку безопасности (RuStore сканирует APK).

### 8.5. Оплата Pro — API Точка (199 ₽/мес)

**На этапе IP без HTTPS:** webhook Точки, скорее всего, **не заработает** (нужен публичный HTTPS). На первом тесте достаточно **rewarded ads**; Точку подключить после домена или временного туннеля (ngrok / cloudflare tunnel).

Когда будет HTTPS:

1. Подключить эквайринг в [Точка Банк](https://tochka.com/) (ИП уже есть).
2. На backend: `POST /api/v1/payments/tochka/create` — платёжная ссылка с `device_id` в metadata.
3. Страница `/pay` — редирект на ссылку Точки.
4. Webhook Точки → `POST /api/v1/payments/tochka/webhook` → поле `pro_until` в таблице `devices`.
5. В приложении: кнопка «Pro 199 ₽» на E4 → Custom Tabs.

### 8.6. Реклама (rewarded video)

1. Зарегистрироваться в [Рекламной сети Яндекса](https://partner.yandex.ru/) / кабинет мобильной рекламы.
2. Создать приложение, получить **App ID**.
3. Создать ad unit типа **Rewarded**.
4. Подключить Yandex Mobile Ads SDK в Android.
5. Показывать только на экране `limit_hit` (не баннеры на главном — не отвлекают от теста).

### 8.7. Модерация и публикация

1. Отправить на модерацию.
2. Типичные причины отклонения: нет политики конфиденциальности, вводящее описание («−10 кг за неделю»), краш при запуске.
3. После одобрения — приложение в каталоге RuStore.

### 8.8. Выплаты

- Привязать расчётный счёт ИП в кабинете RuStore.
- Выплаты по графику RuStore (уточнить в оферте на момент запуска).
- На тесте с 1–5 подписками важен факт транзакции, не объём выплат.
